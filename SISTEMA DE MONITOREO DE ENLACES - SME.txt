OBJETIVO

Desarrollar una aplicación profesional para monitorear automáticamente la disponibilidad de enlaces de red pertenecientes a unidades institucionales.

El sistema deberá funcionar completamente de forma automática.

El usuario nunca iniciará un monitoreo.

El sistema deberá monitorear permanentemente todas las unidades registradas.

Cuando una unidad presente una falla confirmada, el sistema deberá generar automáticamente un ticket.

Cuando el enlace se recupere, el sistema deberá cerrar automáticamente el ticket.

Toda la información deberá almacenarse localmente.

El sistema deberá funcionar aunque no exista conexión a Internet.

No utilizar servicios Cloud.

No utilizar Firebase.

No utilizar Supabase.

No utilizar MongoDB.

No utilizar APIs externas.

No utilizar servicios de monitoreo de terceros.

Toda la lógica deberá ejecutarse localmente.

El monitoreo deberá funcionar las 24 horas del día.

La aplicación será una PWA instalable.

El sistema deberá continuar monitoreando aunque ningún usuario tenga abierta la interfaz.

El proyecto deberá estar preparado para crecer durante muchos años sin requerir cambios importantes en su arquitectura.

Antes de comenzar a desarrollar deberás analizar toda la arquitectura y proponer cualquier mejora que consideres necesaria.

--------------------------------------------------
PROMPT MAESTRO
--------------------------------------------------

A partir de este momento deberás considerar las Partes 1 a la 13 como una única especificación oficial del proyecto.

No omitir funcionalidades.

No simplificar procesos.

No reemplazar tecnologías sin justificarlo.

Antes de generar código, analizar completamente la arquitectura.

Si detectas una mejora importante, proponerla antes de implementarla.

Todo el código deberá cumplir con:

• Clean Architecture

• SOLID

• DRY

• KISS

• Repository Pattern

• Service Pattern

• Separación estricta de responsabilidades.

Generar un proyecto completamente funcional, listo para producción, modular, mantenible y escalable.

Todo el código deberá estar documentado.

Todo el código deberá escribirse en TypeScript.

Toda la interfaz deberá seguir el tema oscuro definido anteriormente.

Toda la información deberá almacenarse en SQLite.

El Servicio Monitor deberá ejecutarse permanentemente como un Servicio de Windows independiente.

La aplicación deberá funcionar como PWA instalable.

No utilizar servicios en la nube.

No utilizar dependencias innecesarias.

Optimizar rendimiento, consumo de memoria y uso de CPU.

El resultado esperado es un sistema profesional de monitoreo de enlaces para uso institucional, preparado para crecer durante años, con una arquitectura limpia, estable y fácil de mantener.
